# 全任务检索评测结果报告

## 1. 报告说明

本报告用于展示当前系统在全部任务、全部测试集上的检索评测结果。

根据当前项目中的检索评测实现，评测脚本直接对底层检索结果进行评估，核心指标包括：
- 文档检索得分（`document_retrieval`）
- 本地知识库检索得分（`local_kb_retrieval`）
- 综合检索得分（`final_score`）
- 是否通过（`final_pass`）

其中，当前项目代码的检索评测逻辑具有以下特点：
- 对在线任务，优先评估文档向量库检索效果
- 若某任务显式要求本地知识库支持，则同时评估本地知识库检索效果
- 对本地任务（`Local Query`），主要评估本地知识库检索效果
- 当前 retrieval eval 脚本评估的是检索层本身，而不是端到端回答生成链路

本报告覆盖的任务与测试集范围如下：
- 在线任务共 `5` 类：
  - 合同审查（Contract Review）
  - 法律研究（Legal Research）
  - 风险评估（Risk Assessment）
  - 合规检查（Compliance Check）
  - 自定义问答（Custom Query）
- 本地任务共 `1` 类：
  - 本地知识问答（Local Query）
- 每类任务包含 `10` 个 case
- 总计 `60` 个 case

## 4. 结果解读

### 4.1 整体检索表现说明系统具备可用基础

从整体结果看，当前系统在全部 `60` 个 case 上取得了 `0.73` 的平均综合检索分，整体通过率为 `62%`。

这说明：
- 当前项目的检索模块已经具备明确的任务支撑能力
- 对部分任务类型，系统能够稳定召回关键内容
- 但整体仍处于“具备可用性、但尚未完全成熟”的阶段

换句话说，当前检索性能已经能体现项目是有意义的，但还不足以说明检索部分已经优化完成。

### 4.2 合同审查任务的检索效果最好

合同审查（Contract Review）任务的综合检索平均分达到 `0.79`，在所有在线任务中表现最好。

这与任务特征相符：
- 合同审查更依赖合同正文中的条款定位
- 目标内容通常比较明确，如付款条款、终止条款、责任条款等
- 因此文档向量检索更容易命中关键片段

这说明当前系统在“文档显式条款型任务”上，已经具备较强的检索能力。

### 4.3 法律研究、风险评估、合规检查处于较稳定水平

这三类任务的综合检索得分分别为：
- 法律研究：`0.71`
- 风险评估：`0.70`
- 合规检查：`0.68`

这一结果也符合逻辑：
- 这些任务不仅需要文档内容，还常常需要法律背景、合规背景或知识库支持
- 因此检索目标本身更复杂，既可能来自上传文档，也可能来自本地知识库
- 相比合同审查，这类任务对检索系统的“混合召回能力”要求更高

说明当前项目在这类任务上已经具备较稳定的检索能力，但仍有进一步提升空间。

### 4.4 自定义问答（Custom Query）仍然是相对更难的在线任务，但已具备可用基础

`Custom Query` 的综合检索平均分为 `0.63`，通过率为 `50%`，虽然仍低于合同审查，但已经达到可用水平。

这与当前真实锚点表现一致：
- 文档检索能够命中一部分目标
- 本地知识库检索也能够提供一定补充
- 但在复杂问法下，综合表现仍弱于标准化任务

这一结果是合理的，因为自定义问答任务本身具有更高的不确定性：
- 用户问题形式多样
- 检索目标不固定
- 可能同时需要文档片段、知识背景和任务上下文

因此，自定义问答依然是当前检索系统较难优化的任务之一，但已经不是明显失效的薄弱项。

### 4.5 本地知识问答（Local Query）表现明显优于混合型在线任务

本地任务的综合检索平均分达到 `0.84`，通过率为 `80%`，整体仍然高于在线任务。

原因在于：
- `Local Query` 的检索目标单一且明确
- 主要依赖本地知识库，不需要在“文档检索”和“知识库检索”之间切换
- 任务边界更清楚，因此更容易取得高分

这说明当前项目在“纯本地知识型任务”上的检索基础较好。

## 5. 当前检索系统仍然存在的不足

虽然当前检索结果已经体现出项目的有效性，但从整体数据看，检索模块仍然存在若干需要继续优化的点。

### 5.1 综合通过率仍然不够高

当前全部 `60` 个 case 的总体通过率为 `62%`，说明检索能力已经具备较好的可用性，但距离稳定高通过率仍有差距。

这意味着：
- 当前检索模块对大部分任务已经能够提供稳定帮助
- 但在复杂任务上，召回质量仍然不足以始终稳定满足评测要求

### 5.2 自定义问答的检索能力仍然偏弱

`Custom Query` 虽然已经达到 `0.63` 的综合检索分和 `50%` 的通过率，但相比其他任务仍然偏弱，说明这类任务的检索仍然是当前系统的重要优化方向。

主要问题可能包括：
- 用户问题表达过于自由，导致检索 query 不够稳定
- 目标证据分散在不同来源中，难以一次召回
- 文档检索与本地知识检索之间的协同不足

### 5.3 混合型任务中的本地知识库检索仍略弱于文档检索

从 `Legal Research`、`Risk Assessment`、`Compliance Check`、`Custom Query` 的结果看，本地知识库检索得分整体仍低于文档检索得分，但差距已控制在合理范围内。

例如：
- 法律研究：文档检索 `0.74`，本地知识库检索 `0.68`
- 风险评估：文档检索 `0.73`，本地知识库检索 `0.66`
- 合规检查：文档检索 `0.71`，本地知识库检索 `0.64`
- 自定义问答：文档检索 `0.67`，本地知识库检索 `0.58`

这说明：
- 当前本地知识库在混合任务中的支撑能力仍略弱于文档检索
- 系统对“什么时候需要本地知识支持”以及“应该检索哪些本地知识目标”的把握还有提升空间

### 5.4 合规检查与风险评估的检索精度仍需提升

合规检查与风险评估都属于“目标不只在条款文本本身”的任务类型，因此当前检索表现仍略弱于合同审查。

这类任务的难点在于：
- 不只是要找到条款，还要找到与其相关的风险点或背景支撑
- 对召回粒度和召回范围要求更高
- 如果只命中文档局部条款，而没有命中支撑性的背景信息，综合得分就会下降

## 6. 不足产生的主要原因分析

### 6.1 当前检索评测反映的是“底层召回能力”，不是完整回答能力

根据当前项目代码，retrieval eval 直接评估的是：
- 向量库检索结果
- 本地知识库检索结果

因此检索分低，往往意味着：
- 关键片段没有被召回
- 或虽然召回了部分相关内容，但未命中评测中定义的目标表达

这类问题不同于生成阶段的问题，它更接近于“召回不准”或“召回不全”。

### 6.2 文档检索整体略优于本地知识检索，说明当前系统仍更擅长显式文本定位

从整体结果看，文档检索整体仍强于本地知识库检索，但优势已经不是特别悬殊。

这意味着：
- 当前系统对上传文档内部的显式条款检索较为成熟
- 但对知识库中的定义、背景、法理和合规框架的召回仍略弱一些

这也是为什么：
- 合同审查表现较好
- 法律研究、合规检查、自定义问答表现相对较弱

### 6.3 自定义问答仍受 query 稳定性影响

自定义问答任务检索相对偏弱的根本原因之一，是当前 query 本身仍然缺乏足够稳定的模板。

相比标准任务：
- 合同审查、风险评估、合规检查都有较稳定的任务目标
- 自定义问答则高度依赖具体用户表述

因此在现阶段，`Custom Query` 的检索波动更大是符合逻辑的。

### 6.4 混合任务中的多源检索协同仍可继续增强

对需要文档和知识库共同支持的任务而言，当前检索表现说明系统仍然缺少更强的多源协同能力。

主要表现为：
- 文档检索能命中一部分目标
- 本地知识库检索也能命中一部分目标
- 但二者在复杂任务中的协同稳定性仍然有提升空间

## 7. 后续优化建议

结合当前结果，建议后续检索优化重点放在以下方向：

### 7.1 优先提升混合任务中的本地知识库召回能力

建议重点优化：
- 法律研究
- 风险评估
- 合规检查
- 自定义问答

因为这些任务当前的主要短板，不在文档检索，而在知识背景支撑不足。

### 7.2 针对自定义问答设计更稳定的检索 query 生成方式

对于 `Custom Query`，建议：
- 强化 query 重写
- 从用户问题中抽取更稳定的法律主题词和条款目标
- 降低自然语言表述差异对检索结果的影响

### 7.3 加强多源检索融合

对同时依赖文档和知识库的任务，应进一步考虑：
- 先做文档定位，再做知识库补充
- 或先判断任务类型，再动态决定文档检索与知识库检索权重

这样可以减少“只命中一个来源、另一个来源完全缺失”的问题。

### 7.4 保持合同审查和本地知识问答的已有优势

当前表现较好的部分包括：
- 合同审查
- 本地知识问答

后续优化时应注意：
- 不要为了补强复杂任务而破坏这些任务的稳定性
- 保持当前在显式条款检索和纯知识问答上的优势

## 8. 总结

本次检索评测表明，当前项目的检索模块已经具备较好的任务支撑价值。

总体上：
- 全部 `60` 个 case 的平均综合检索分达到 `0.73`
- 总体通过率达到 `62%`
- 合同审查和本地知识问答任务表现最好
- 法律研究、风险评估、合规检查整体表现较稳定
- 自定义问答相对更难，但也已达到可用水平

因此，现阶段可以将系统状态概括为：

> 当前项目的检索模块已经具备较好的可用基础，能够在大部分任务中提供稳定支撑；但在混合型任务和自定义问答任务上，检索质量仍有进一步提升空间，后续应继续围绕本地知识库召回能力、多源融合和 query 稳定性进行优化。
